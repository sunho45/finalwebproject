# finalwebproject 프로젝트 계획서

## 1. 프로젝트 주제

본 프로젝트는 사용자가 공부 계획을 등록하고 관리할 수 있는 웹 기반 학습 플래너를 제작하는 것을 목표로 한다. 사용자는 해야 할 공부 내용, 과목, 날짜, 예상 공부 시간을 입력할 수 있으며, 등록된 계획의 완료 여부를 체크하고 삭제할 수 있다.

단순히 화면에서만 동작하는 할 일 목록이 아니라, 백엔드 서버와 데이터베이스를 연결하여 실제 데이터가 저장되는 풀스택 웹 서비스를 구현한다.

## 2. 프로젝트 목표

이 프로젝트의 주요 목표는 다음과 같다.

- React를 활용한 사용자 화면 구현
- Express를 활용한 백엔드 API 서버 구현
- PostgreSQL DBMS를 이용한 공부 계획 데이터 저장
- WebStorage를 활용한 익명 사용자 ID, 사용자 설정, 작성 중인 입력값 저장
- Nginx reverse proxy를 이용한 프론트엔드와 백엔드 요청 분리
- Docker Compose를 활용한 통합 실행 환경 구성
- Render를 이용한 배포 환경 구성

## 3. 개발 범위

본 프로젝트에서 구현할 기능은 다음과 같다.

| 구분 | 구현 계획 |
| --- | --- |
| 계획 등록 | 제목, 과목, 날짜, 예상 공부 시간을 입력해 계획 저장 |
| 계획 조회 | 저장된 공부 계획 목록 표시 |
| 완료 처리 | 각 계획의 완료 여부 변경 |
| 계획 삭제 | 필요 없는 계획 삭제 |
| 통계 표시 | 전체 계획 수, 완료 계획 수, 누적 공부 시간 표시 |
| 필터 기능 | 전체, 진행 중, 완료 항목 구분 표시 |
| 테마 설정 | 라이트/다크 테마 전환 |
| 입력값 보존 | 작성 중인 계획 초안을 브라우저에 임시 저장 |
| 사용자별 데이터 | 브라우저별 익명 사용자 ID를 기준으로 계획 분리 |

## 4. 프론트엔드 계획

프론트엔드는 React와 Vite를 사용하여 구현한다. React 컴포넌트에서 입력 폼, 계획 목록, 통계 영역, 필터 버튼, 테마 전환 기능을 구성한다.

화면은 사용자가 바로 공부 계획을 추가하고 확인할 수 있도록 단순하고 직관적인 구조로 설계한다. 주요 상태 관리는 React의 `useState`, `useEffect`, `useMemo`를 사용한다.

프론트엔드 주요 구현 항목은 다음과 같다.

- 공부 계획 입력 폼
- 계획 목록 렌더링
- 완료 상태 표시
- 삭제 버튼
- 진행률 표시
- 전체, 진행 중, 완료 필터
- 라이트/다크 테마 전환
- 반응형 스타일 적용

## 5. 백엔드 계획

백엔드는 Node.js와 Express를 사용하여 REST API 서버로 구현한다. 프론트엔드에서 요청을 보내면 서버가 요청을 처리하고 PostgreSQL 데이터베이스와 연동한다.

백엔드에서는 계획 목록 조회, 계획 등록, 완료 상태 수정, 계획 삭제 기능을 API로 제공한다. 또한 서버 상태 확인을 위한 health check API를 제공하여 배포 환경에서 서비스 상태를 확인할 수 있도록 한다.

## 6. API 설계 계획

이 프로젝트에서 사용하는 API는 외부 API가 아니라 Express 서버에서 직접 구현하는 내부 REST API이다. 클라이언트는 `localStorage`에 저장된 익명 사용자 ID를 `X-Planner-User` 헤더로 전달하고, 서버는 이 값을 기준으로 해당 사용자 데이터만 처리한다.

| Method | Endpoint | 기능 |
| --- | --- | --- |
| GET | `/api/health` | 서버 상태 확인 |
| GET | `/api/tasks` | 계획 목록과 통계 조회 |
| POST | `/api/tasks` | 새 계획 등록 |
| PATCH | `/api/tasks/:id` | 완료 상태 변경 |
| DELETE | `/api/tasks/:id` | 계획 삭제 |

API 동작 흐름은 다음과 같이 설계한다.

```text
React 화면 -> 사용자 ID 포함 API 요청 -> Express 서버 -> PostgreSQL 사용자별 데이터 처리 -> JSON 응답 -> React 화면 갱신
```

## 8. WebStorage 활용 계획

브라우저 저장소는 사용자 편의성을 높이기 위해 사용한다.

- `localStorage`: 브라우저별 익명 사용자 ID 저장
- `localStorage`: 테마, 필터, 기본 집중 시간 저장
- `sessionStorage`: 작성 중인 공부 계획 초안 저장

`localStorage`는 브라우저를 닫았다가 다시 열어도 값이 유지되므로 익명 사용자 ID와 사용자 설정 저장에 사용한다. 이 사용자 ID는 로그인 계정은 아니지만, 같은 브라우저에서 작성한 계획만 다시 불러오기 위한 구분값으로 사용한다. `sessionStorage`는 현재 탭에서만 유지되므로 작성 중인 입력값을 임시 저장하는 데 사용한다.

## 9. DBMS 설계 계획

DBMS는 PostgreSQL을 사용한다. 공부 계획 데이터는 `tasks` 테이블에 저장한다.

`tasks` 테이블의 주요 컬럼은 다음과 같이 설계한다.

| 컬럼 | 설명 |
| --- | --- |
| `id` | 계획 고유 ID |
| `user_key` | 브라우저별 익명 사용자 ID |
| `title` | 공부 계획 제목 |
| `subject` | 과목 |
| `due_date` | 계획 날짜 |
| `minutes` | 예상 공부 시간 |
| `done` | 완료 여부 |
| `created_at` | 생성 시각 |

기본적인 데이터 무결성을 위해 사용자 ID, 제목, 과목, 날짜, 공부 시간은 필수값으로 설정하고, 공부 시간은 0보다 큰 값만 저장되도록 제한한다. API는 `user_key`를 기준으로 조회, 수정, 삭제 범위를 제한하여 다른 사용자의 계획이 섞이지 않도록 한다.

## 10. Reverse Proxy 계획

로컬 실행 환경에서는 Nginx를 reverse proxy로 사용한다. 사용자는 하나의 주소인 `http://localhost:8080`으로 접속하고, Nginx가 요청 경로에 따라 프론트엔드와 백엔드로 요청을 분리한다.

- `/api/` 요청: Express 서버로 전달
- 그 외 요청: React 클라이언트로 전달

이 구성을 통해 프론트엔드와 백엔드를 분리하면서도 사용자는 하나의 서비스처럼 이용할 수 있다.

## 11. Docker 실행 계획

Docker Compose를 사용하여 프로젝트 실행 환경을 구성한다. 하나의 명령으로 데이터베이스, 백엔드 서버, 프론트엔드 클라이언트, Nginx 프록시가 함께 실행되도록 한다.

서비스 구성 계획은 다음과 같다.

| 서비스 | 역할 |
| --- | --- |
| `db` | PostgreSQL 데이터베이스 |
| `server` | Express API 서버 |
| `client` | React 정적 파일 제공 |
| `nginx` | reverse proxy |

로컬 실행 명령은 다음과 같이 계획한다.

```bash
docker compose up --build
```

## 12. 배포 계획

배포는 Render를 기준으로 구성한다. Render Web Service는 Docker 런타임으로 실행하며, 컨테이너 안에서 Nginx와 Express 서버를 함께 실행한다. Nginx는 외부 요청을 받는 reverse proxy 역할을 하고, Express는 컨테이너 내부 포트에서 API를 처리한다. PostgreSQL 데이터베이스도 Render에서 함께 생성하여 연결한다.

배포 과정은 다음과 같다.

1. GitHub 저장소에 프로젝트 업로드
2. Render에서 Blueprint 생성
3. `render.yaml`을 기준으로 Docker Web Service와 PostgreSQL 생성
4. Docker 이미지 안에서 React 빌드, Nginx 설정, Express 서버 실행
5. `/api/health`를 통해 배포 후 서버 상태 확인

## 13. 기대 효과

이 프로젝트를 통해 프론트엔드, 백엔드, 데이터베이스, 프록시, Docker, 배포까지 연결된 웹 서비스 구조를 경험할 수 있다. 또한 공부 계획 관리라는 실용적인 주제를 통해 사용자가 직접 데이터를 입력하고 관리하는 CRUD 웹 애플리케이션의 전체 흐름을 이해할 수 있다.

최종적으로는 단순한 화면 구현을 넘어 실제 저장소와 배포 환경까지 고려한 학습 플래너 웹 서비스를 완성하는 것을 목표로 한다.
